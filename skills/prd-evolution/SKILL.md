# PRD Evolution - PRD 树状版本管理器

> **角色**：PRD 版本管理专家
>
> **核心价值**：管理 PRD 的完整生命周期，处理需求变更时的覆盖/扩展逻辑，维护清晰的版本树

---

## 🎯 核心概念：PRD 树

```
prd-root (初始 PRD)
├── prd-v1.0 (已实现)
│   ├── feature-A (已实现)
│   └── feature-B (已实现)
├── prd-v1.1 (扩展)
│   └── feature-C (新功能，待实现)
└── prd-v2.0 (覆盖)
    ├── feature-A-v2 (覆盖 feature-A)
    └── feature-D (新功能)
```

### 节点状态

| 状态 | 图标 | 含义 |
|------|------|------|
| 草稿 | 🟡 | 刚创建，待确认 |
| 已确认 | 🟢 | 需求已确认，待实现 |
| 已实现 | 🔵 | 代码已完成 |
| 测试中 | 🟣 | 实现完成，测试中 |
| 已废弃 | ⚪ | 被新版本覆盖 |
| 已合并 | ✅ | 已合并到主线 |

---

## 🌳 树状结构规则

### 规则 1：功能描述不清楚 → 扩展

**场景**：功能 A 已实现，但发现描述不够详细，需要补充细节

**操作**：
```
feature-A (已实现)
└── feature-A-detailed (扩展，待实现)
    └── 继承 feature-A 的实现，添加新细节
```

**实现策略**：
- 保留原有功能
- 在新分支上添加细节
- 完成后合并或并存

### 规则 2：功能未实现 → 直接覆盖

**场景**：功能 B 还未实现（🟡/🟢），用户修改了需求

**操作**：
```
feature-B-v1 (🟡 草稿) ──→ feature-B-v2 (🟡 草稿)
     ↓                           ↓
   [废弃]                      [新需求]
```

**实现策略**：
- 废弃旧版本
- 新版本继承 ID 和位置
- 直接按新版本实现

### 规则 3：功能已实现但需求变更 → 扩展新版本

**场景**：功能 C 已实现（🔵），但用户要求修改

**操作**：
```
feature-C-v1 (🔵 已实现)
└── feature-C-v2 (🟡 草稿，待实现)
    └── 基于 v1 修改
```

**实现策略**：
- 保留 v1 实现（可能回滚）
- v2 基于 v1 代码修改
- 完成后 v1 标记为废弃

## 📋 工作流程

### 场景 1：更新已有 PRD

```
用户：更新 PRD prd-20260323-143022-a3f7，添加搜索功能

AI：
1. 读取原 PRD
2. 判断原 PRD 状态
3. 根据状态决定策略
4. 创建新版本或扩展节点
5. 更新树状结构
6. 返回新版本 PRD ID
```

### 场景 2：修改功能需求

```
用户：修改功能"任务创建"，添加优先级字段

AI：
1. 找到功能节点
2. 检查功能状态
3. 状态=🔵已实现 → 创建扩展节点
4. 状态=🟡草稿 → 直接覆盖
5. 更新 PRD 文档
6. 更新树结构图
```

### 场景 3：废弃功能

```
用户：废弃功能"邮件通知"

AI：
1. 找到功能节点
2. 标记为 ⚪ 已废弃
3. 添加废弃原因
4. 更新树结构
5. 如果有代码，标记为待删除
```

## ⚠️ 注意事项

1. **自动策略选择**: 根据状态自动选择最佳策略，但用户可以强制指定
2. **变更记录**: 所有变更必须记录到 tree.md 的变更历史中
3. **代码同步**: 如果功能已实现，修改 PRD 时需要评估代码影响
4. **版本号**: 遵循语义化版本（major.minor.patch）
5. **备份**: 废弃的 PRD 移动到 archive/ 目录，不直接删除
