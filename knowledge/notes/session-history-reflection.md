## 🎯 最佳实践（5 个领域）

### 1. 键盘配置（keyd）

**原则**：
1. 修改前先 `git status` 检查当前状态
2. 参考历史 git 记录中的成功配置
3. 修改后提交到 git，便于回滚
4. ROG 特殊按键（m4, f4）需要两套键盘 ID 配置

**验证命令**：
```bash
sudo keyd check && sudo keyd reload
ps aux | grep keyd
```

---

### 2. OpenClaw 部署

**原则**：
1. 服务用户隔离：openclaw 服务以独立用户运行
2. 配置目录必须在 `/home/openclaw/.openclaw/`
3. 禁止让 openclaw 用户访问 `/home/throokie/` 目录

**验证命令**：
```bash
ps aux | grep openclaw  # 检查运行用户
sudo journalctl -u openclaw | grep -i EACCES  # 检查权限错误
```

---

### 3. MCP 工具使用

**原则**：
1. MCP 优先，Skill 后备
2. 失败必须透明化，禁止沉默跳过
3. 有 5 个 MCP 可用：context7, sequential-thinking, github, chrome-devtools, weixin_search_mcp

**决策流程**：
```
用户请求 → 识别意图 → 优先 MCP → MCP 无法满足 → 降级 Skill
```

---

### 4. 问题排查

**原则**：
1. 先读日志，再分析
2. 权限错误优先级最高（EACCES）
3. 系统性检查，不瞎猜

**调试优先级**：
```
1. 权限错误 (EACCES) ← 最高优先级，看到就先解决
2. 服务运行状态 (systemctl status | ps aux)
3. 配置语法验证 (cat | jq .)
4. 网络/连接问题 (ss -tlnp | curl -v)
5. 业务逻辑 ← 只有以上都正常才开始分析
```

---

### 5. Skill 管理

**原则**：
1. 优先用 MCP，MCP 无法满足时用 Skill
2. Skill 目录用软连接指向 `~/src/user-scripts/skills/`
3. 新 Skill 必须是 `.md` 格式或脚本形式

## 🔄 后续自动化建议

### 建议 1: 会话结束自动反思（❌ 不推荐）

**方案**：每次会话结束时自动执行 `insight-capture`

**问题**：
- 会产生大量低价值记录
- 用户可能不想记录每次对话

---

### 建议 2: 周期性回顾（✅ 推荐）

**方案**：每周/每月执行一次深度反思

**触发方式**：用户主动说"周总结"或"月度回顾"

**执行内容**：
1. 读取本周历史对话
2. 识别新的高频问题
3. 更新错误模式库
4. 提炼新的最佳实践

---

### 建议 3: 问题驱动反思（✅ 推荐）

**方案**：当遇到重复问题时，自动触发反思

**触发条件**：
- 同一主题出现 3 次+
- 同一错误模式重复踩坑

**执行内容**：
1. 搜索历史对话
2. 分析为什么重复出现
3. 更新文档或建立新规则
