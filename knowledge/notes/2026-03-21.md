## [INS-031] clipboard-compact v2.0 URL 修复重构

### 问题根因
**现象**：用户反馈 URL 中的问号被处理没了，换行变空格导致 URL 截断无法访问

**示例**：
```
输入：https://example.com/api?
key=1

v1.0 处理:
1. merge_to_one_line → "https://example.com/api? key=1"  ← ? 后多了空格
2. fix_url → 移除空格但 URL 结构已破坏

期望输出：https://example.com/api?key=1
```

### 核心教训

**教训 1**：先合并再修复的顺序是错误的
- 合并操作将换行变成空格，破坏了 URL 的原始结构
- 后续修复无法恢复已被破坏的位置信息

**教训 2**：URL 修复应该一次性完成
- 直接用 regex 匹配完整 URL（包括内部可能的空格）
- 在匹配范围内移除所有空白字符
- 避免多步骤处理导致的信息丢失

**教训 3**：heredoc + Python 的参数传递陷阱
- `python3 << 'EOF'` 会占用 stdin
- Python 的 `sys.stdin.read()` 会读取 heredoc 内容，不是外部变量
- 正确方式：使用环境变量 `CONTENT="value" python3 << 'EOF'` + `os.environ.get('CONTENT')`

### 解决方案模式

```bash
fix_wrapped_url() {
    CONTENT="$1" python3 << 'PYTHON_SCRIPT'
import os
text = os.environ.get('CONTENT', '')
# 直接用 regex 匹配并清理，避免多步骤处理
url_pattern = r"https?://[a-zA-Z0-9._~:/?#\[\]@!$&'()*+,;=%\-\s]+"
# ... 处理逻辑 ...
PYTHON_SCRIPT
}
```

### 验证清单

修复后的 URL 必须通过以下测试：
- [ ] 查询参数完整 (`?key=123&token=abc`)
- [ ] 编码字符保留 (`%20` 不被解码)
- [ ] Fragment 保留 (`#section`)
- [ ] URL 后标点被移除 (`访问 xxx.com。` → `访问 xxx.com`)
- [ ] 多 URL 混合文本正确处理

## [CODING-012] 重构四步法实践

### 方法论
按照用户提供的提示词模板执行重构：

1. **功能需求拆解** - 忘掉现有代码，从核心目标拆解 3-5 个 Use Cases
2. **现有代码缺陷分析** - 指出致命缺陷、为什么满足不了需求
3. **关键实现逻辑设计** - 伪代码/流程图描述数据流、核心算法、边界情况
4. **核心骨架实现** - 清晰的函数结构、核心算法、中文注释

### 应用效果
本次 clipboard-compact v2.0 重构：
- 第一步识别出 4 个 Use Cases（URL 完整性保护、终端换行识别、类型检测、边界处理）
- 第二步找出 4 个致命缺陷（merge 破坏结构、regex 贪婪、?=&被错误处理、fragment 未处理）
- 第三步设计了 URL 精确匹配 regex + 环境变量传参方案
- 第四步实现了核心 fix_wrapped_url 函数

### 以后应用
对于"实现逻辑较差、无法完全满足功能"的代码重构任务，主动使用四步法：
1. 先拆解需求（不急着看代码）
2. 再分析缺陷（对照需求找问题）
3. 设计逻辑（伪代码/流程图）
4. 最后实现（核心骨架 + 注释）
