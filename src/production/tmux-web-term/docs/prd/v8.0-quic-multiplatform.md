## 2. 产品概述

### 2.1 项目背景

tmux-web-term 是一个基于 WebSocket 的 tmux 远程访问工具，允许用户通过浏览器管理服务器上的 tmux 会话。当前版本（v7）已实现：
- WebSocket 实时终端通信
- 多设备同屏（tmux 会话共享）
- 内网穿透支持（Tailscale/ZeroTier）
- PWA 离线缓存、asciinema 录制

**核心痛点**：
- WebSocket/HTTP 基于 TCP，握手开销大（1-RTT 或更多）
- 不支持连接迁移——网络切换（WiFi ↔ 4G）时连接断开
- 移动网络下延迟高、抗丢包能力差
- 浏览器沙箱限制无法使用更高效的传输协议

### 2.2 产品目标

打造**"不妥协"的高性能远程终端工具**：
1. **极致性能**：0-RTT 握手，连接延迟 < 100ms
2. **连接稳定**：WiFi/4G/5G 无缝切换，连接不中断
3. **弱网可用**：高丢包环境下仍保持流畅体验
4. **全平台覆盖**：Web + iOS + Android + Desktop（exe/bin）
5. **向后兼容**：Web 端保持现有体验，原生 App 享受高性能

### 2.3 目标用户

| 用户类型 | 特征 | 核心需求 |
|----------|------|----------|
| 移动开发者 | 经常外出，需要手机管理服务器 | 移动网络下稳定、低延迟 |
| DevOps 工程师 | 多设备切换，7x24 待命 | 连接不中断，会话持久化 |
| 远程工作者 | 跨地域协作，网络环境复杂 | 弱网可用，抗丢包 |
| 极客用户 | 追求极致性能，愿意尝试新技术 | 0-RTT、连接迁移、原生体验 |

### 3.2 详细功能描述

#### F-P0-001 QUIC 服务端

**需求描述**：
服务端需要支持 QUIC/HTTP3 协议，作为原生 App 的主要通信方式。

**技术要点**：
- 使用 `aioquic` 库实现 QUIC 服务器
- 支持 0-RTT 握手（会话恢复）
- 支持连接迁移（Connection Migration）
- 内置 TLS 1.3 加密

**验收标准**：
- [ ] QUIC 服务器可独立启动，监听 UDP 端口（默认 9443，避免与现有端口冲突）
- [ ] 支持 0-RTT，重复连接握手延迟 < 100ms
- [ ] 支持多路复用流（Bidirectional Streams）
- [ ] 单服务器支持 1000+ 并发 QUIC 连接

#### F-P0-002 WebSocket 降级

**需求描述**：
实现双协议栈，当 QUIC 不可用时（防火墙阻断、浏览器不支持），自动降级到 WebSocket。

**技术要点**：
- 协议协商：客户端优先尝试 QUIC，失败后降级
- 统一会话管理：WebSocket 和 QUIC 连接同一 tmux 会话
- 透明降级：用户无需手动切换

**验收标准**：
- [ ] WebSocket 服务器保持现有功能
- [ ] 协议协商自动完成，无需用户干预
- [ ] 降级过程连接不中断（或快速重连）

#### F-P0-003 协议抽象层

**需求描述**：
设计协议抽象层，统一封装 QUIC 和 WebSocket，业务逻辑层无感知底层协议差异。

**接口设计**：
```python
class Connection(ABC):
    @abstractmethod
    async def send(self, data: bytes) -> None: ...

    @abstractmethod
    async def recv(self) -> bytes: ...

    @abstractmethod
    async def close(self) -> None: ...

    @property
    @abstractmethod
    def protocol(self) -> ProtocolType: ...

class QUICConnection(Connection): ...
class WebSocketConnection(Connection): ...
```

**验收标准**：
- [ ] 业务代码仅依赖 Connection 接口
- [ ] 新增协议无需修改业务代码
- [ ] 连接池管理统一处理

#### F-P0-004 Flutter 移动端

**需求描述**：
开发 iOS/Android 原生 App，使用 Flutter 框架，与移动端代码高度复用。

**技术要点**：
- Flutter + Dart FFI 调用 Rust QUIC 客户端
- 使用 `flutter_xterm` 或自研终端渲染组件
- 共享核心逻辑：协议、会话管理、配置

**验收标准**：
- [ ] iOS App 可在 App Store 发布
- [ ] Android App 可在 Google Play/国内商店发布
- [ ] 代码复用率 > 90%（与桌面端）
- [ ] 性能：启动时间 < 3s，终端渲染 60fps

#### F-P0-005 会话同步

**需求描述**：
实现统一会话 ID，不同协议（WebSocket/QUIC）、不同平台（Web/移动端）可连接同一 tmux 会话。

**技术要点**：
- 会话 ID 作为唯一标识
- 会话管理器维护协议到会话的映射
- 支持多客户端同时连接同一会话（tmux 原生支持）

**验收标准**：
- [ ] Web 端和移动端可同时连接同一 tmux 会话
- [ ] 会话状态实时同步（输入输出）
- [ ] 客户端断开重连后恢复同一会话

#### F-P1-003 Flutter Desktop

**需求描述**：
开发 Windows/Linux/macOS 桌面端应用，使用 Flutter Desktop，打包为 exe/bin。

**技术要点**：
- Flutter Desktop 嵌入原生窗口
- 系统托盘/菜单栏支持
- 快捷键支持（Ctrl/Cmd + 组合键）

**验收标准**：
- [ ] Windows 安装包（.exe）
- [ ] Linux 可执行文件（.AppImage/.deb）
- [ ] macOS 应用（.dmg/.app）
- [ ] 与移动端代码复用率 > 95%

### 5.4 协议设计

#### 5.4.1 QUIC 协议帧格式

```
Frame Structure:
┌──────────┬──────────┬──────────┬─────────────────┐
│  Type    │  Length  │  Payload │  Checksum       │
│  (1 byte)│ (4 bytes)│  (var)   │  (4 bytes)      │
└──────────┴──────────┴──────────┴─────────────────┘

Frame Types:
0x01 - INIT       - 初始化连接，携带认证信息
0x02 - DATA       - 终端数据
0x03 - RESIZE     - 终端尺寸调整
0x04 - HEARTBEAT  - 心跳保活
0x05 - CLOSE      - 关闭连接
0x06 - ERROR      - 错误通知
```

#### 5.4.2 连接流程

```
Client                                    Server
  │                                         │
  ├─ QUIC Handshake (0-RTT if resumption) ─>│
  │                                         │
  ├─ INIT Frame (token, session_id) ───────>│
  │                                         │
  │<─────────── INIT-ACK Frame ─────────────┤
  │                                         │
  ├─ DATA Frame (user input) ──────────────>│
  │                                         │
  │<────────── DATA Frame (output) ─────────┤
  │                    ...                  │
```

| 开发周期延长 | 中 | 中 | 严格 MVP 范围控制，优先 P0 功能 |
