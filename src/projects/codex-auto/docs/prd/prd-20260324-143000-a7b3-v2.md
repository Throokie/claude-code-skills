## 2. 产品概述

### 2.1 产品背景

Codex 是 OpenAI 提供的代码生成模型服务，官方 API 存在以下痛点：
- **访问限制**：部分地区或场景访问受限
- **配额限制**：单个账号有使用配额限制，高并发场景不够用
- **账号管理**：批量管理多个账号繁琐，健康状态难以监控

本项目旨在通过多账号池化管理 + 反代技术，构建一个高可用的免费 API 池服务。

### 2.2 产品目标

- **P0**: 实现 Codex 反代服务，转换为标准 OpenAI API 格式
- **P0**: 实现账号状态管理与多账号负载均衡
- **P1**: 实现账号自动注册机制（域名池+接码服务）
- **P1**: 实现账号健康监控与自动替换机制
- **P1**: 提供管理后台，可视化监控 API 池状态
- **P2**: 实现用户配额管理和访问控制
- **P2**: 实现日志审计和统计分析

### 2.3 目标用户

| 用户类型 | 特征 | 主要需求 |
|---------|------|---------|
| 开发者 | 需要代码补全/生成功能 | 稳定、低延迟的 API 服务 |
| AI 应用构建者 | 集成 Codex 到自有产品 | 标准 API 格式、高可用性 |
| 运维人员 | 管理API池健康状态 | 监控告警、账号管理 |

### 2.4 合规性声明

> ⚠️ **重要声明**：本项目仅供学习研究使用，请遵守以下原则：
> 1. 遵守 OpenAI/Codex 服务条款
> 2. 遵守当地法律法规
> 3. 禁止用于商业盈利目的
> 4. 用户自行承担使用风险

### 3.2 详细功能描述

#### F-003: Codex 反代服务

**用户故事**: 作为开发者，我想要通过标准 API 格式调用 Codex，以便集成到现有工具

**验收标准**:
- [ ] 支持 `/v1/chat/completions` 标准端点
- [ ] 支持流式响应 (SSE)
- [ ] 支持模型列表查询 `/v1/models`
- [ ] 错误处理：将 Codex 错误转换为标准 OpenAI 错误格式
- [ ] P95响应延迟 < 5秒（不含AI生成时间）

**输入/输出**:
- 输入: OpenAI API 格式请求（含API Key认证头）
- 输出: OpenAI API 格式响应（流式或非流式）

**边界情况**:
- Codex服务不可用: 返回503状态码，携带错误信息
- 无效API Key: 返回401状态码
- 请求超时: 30秒后返回504状态码
- 流式中断: 优雅关闭SSE连接

#### F-002: 账号状态管理

**用户故事**: 作为运维人员，我想要管理 API 池中的账号，以便维护服务可用性

**验收标准**:
- [ ] 支持账号CRUD操作（添加、删除、编辑、查询）
- [ ] 状态追踪：正常、限速、封禁、失效、错误
- [ ] 配额追踪：记录每个账号的剩余额度
- [ ] 批量导入/导出账号（JSON/CSV格式）

**账号状态定义**:

| 状态 | 说明 | 自动切换策略 |
|------|------|--------------|
| active | 正常可用 | - |
| rate_limited | 触发速率限制 | 冷却1小时后重试 |
| suspended | 被平台封禁 | 自动标记，需人工确认 |
| expired | 额度耗尽 | 自动下线 |
| error | 其他错误 | 重试3次后标记 |

**输入/输出**:
- 输入: 账号信息（邮箱、密码、状态等）
- 输出: 操作结果、账号列表

**边界情况**:
- 重复添加: 检查邮箱唯一性，返回冲突错误
- 删除正在使用的账号: 提示风险， graceful下线
- 批量导入部分失败: 记录失败原因，支持部分导入

#### F-004: 多账号负载均衡

**用户故事**: 作为 API 使用者，我想要高可用的服务，以便在部分账号失效时仍能正常使用

**验收标准**:
- [ ] 支持轮询负载均衡策略（MVP）
- [ ] 支持权重策略（后续版本）
- [ ] 健康检查：定期检测账号可用性
- [ ] 故障转移：账号失效时自动切换到可用账号
- [ ] 99%+ 服务可用性

**路由策略配置**:

```yaml
load_balance:
  strategy: round_robin  # round_robin | weighted
  health_check:
    enabled: true
    interval: 60s
    timeout: 10s
  retry:
    max_attempts: 3
    backoff: exponential
```

**边界情况**:
- 所有账号不可用: 返回503，触发紧急告警
- 单账号rate limit: 自动标记冷却，切换账号
- 网络超时: 重试3次后标记为error

#### F-001: 账号自动注册

**用户故事**: 作为管理员，我想要系统自动创建 Codex 账号，以便扩充 API 池容量

**验收标准**:
- [ ] 支持配置域名池（用户输入的域名列表）
- [ ] 支持配置接码平台 API（如 5sim、smshub、sms-activate等）
- [ ] 自动化注册流程：生成邮箱 → 接码获取手机号 → 注册账号 → 保存凭证
- [ ] 注册成功率监控，失败自动重试
- [ ] 支持注册速率限制，防止被封

**注册频率控制**:
- 每账号间隔：最少1小时
- 每日上限：5个账号
- 失败重试：指数退避（1min → 5min → 15min → 1hour）

**输入/输出**:
- 输入: 域名池配置、接码平台配置、注册数量
- 输出: 注册成功的账号列表（保存到数据库）

**边界情况**:
- 接码平台无可用号码: 等待并重试，或切换到备用接码平台
- 注册触发风控: 增加延时、更换IP代理、降低频率
- 域名被拉黑: 标记域名不可用，从池中移除

#### F-005: API 网关

**用户故事**: 作为服务提供者，我想要控制 API 访问，以便管理服务质量和成本

**验收标准**:
- [ ] API Key 认证机制
- [ ] 请求限流（Rate Limiting）：按用户/全局配置
- [ ] IP 白名单/黑名单

**限流策略**:

| 级别 | 策略 | 示例 |
|------|------|------|
| 全局 | 总 QPS 限制 | 100 req/s |

**边界情况**:
- 超过限流: 返回429状态码，携带Retry-After头
- 无效API Key: 返回401状态码
- IP被封禁: 返回403状态码

#### F-006: 健康监控告警

**用户故事**: 作为运维人员，我想要及时发现服务异常，以便快速响应修复

**验收标准**:
- [ ] 账号健康度仪表盘（可用率、响应时间、错误率）
- [ ] 告警机制：账号大面积失效、请求成功率下降、响应时间异常
- [ ] 告警渠道：Webhook、Telegram
- [ ] 自动恢复：部分场景自动尝试恢复（如 rate limit 冷却）

**监控指标**:

```yaml
metrics:
  business:
    - 可用账号数 / 总账号数
    - 请求成功率 / 失败率
    - 平均响应时间 / P95 / P99
    - 每分钟请求数(QPS)
  system:
    - 服务内存/CPU使用率
    - 数据库连接数
    - 缓存命中率

alert_thresholds:
  critical: 可用账号数 < 5
  warning: 成功率 < 95% 或 P95延迟 > 10s
```

**边界情况**:
- 告警渠道不可用: 尝试备用渠道，记录到本地日志
- 大面积账号失效: 触发紧急告警，自动暂停注册
- 误报频繁: 告警静默期，避免告警风暴

## 5. 风险识别与应对

### 5.1 风险矩阵

| 风险类别 | 风险项 | 概率 | 影响 | 风险等级 |
|----------|--------|------|------|----------|
| 合规 | 违反Codex ToS | 中 | 极高 | 🔴 |
| 账号安全 | 批量注册触发风控 | 高 | 极高 | 🔴 |
| 外部依赖 | 接码平台服务中断 | 中 | 高 | 🟡 |
| 资源 | 域名被拉黑/耗尽 | 中 | 高 | 🟡 |
| 运维 | 单点故障 | 中 | 中 | 🟡 |
| 技术 | Codex接口变更 | 低 | 高 | 🟢 |
| 安全 | 凭证泄露 | 低 | 极高 | 🟡 |

### 5.2 风险应对策略

#### 🔴 账号封禁风险

**预防措施**:
- 注册频率控制：每账号间隔1小时，每日上限5个
- IP代理轮换：使用住宅代理池，每个注册使用不同IP
- 浏览器指纹模拟：使用Puppeteer模拟真实用户
- 多域名分散：避免单域名大量注册
- 冷却机制：账号注册后静默24小时再使用

**检测机制**:
- 可用账号数实时监控
- 注册失败率监控
- 账号状态变更日志

**应对措施**:
- 可用账号 < 阈值：自动暂停注册，触发告警
- 人工介入审查注册策略
- 备用账号池储备

#### 🔴 合规风险

**缓解措施**:
- 明确标注仅供学习研究使用
- 限制单用户请求频率，避免商业滥用
- 鼓励用户私有部署
- 选择合适开源协议明确责任边界

### 5.3 应急响应预案

```yaml
emergency_response:
  账号大面积封禁:
    trigger: 可用账号数 < 总数的30%
    actions:
      - 自动暂停自动注册
      - 发送紧急告警（多渠道）
      - 切换至只读模式（仅查询不消耗账号）
      - 记录事件日志
    recovery: 人工介入审查后恢复

  接码平台失效:
    trigger: 连续5次注册失败
    actions:
      - 切换到备用接码平台
      - 发送告警通知
      - 暂停注册任务
    recovery: 平台恢复后自动恢复

  服务不可用:
    trigger: 所有账号不可用或核心服务故障
    actions:
      - 返回503服务不可用
      - 触发紧急告警
      - 尝试自动恢复（重启服务）
    recovery: 人工排查修复
```

interface Account {

### 9.1 术语表

| 术语 | 定义 |
|------|------|

| API 池 | 多个 API 账号/端点的集合，提供统一访问入口 |
| 接码平台 | 提供虚拟手机号接收短信验证码的服务 |
| 域名池 | 用于注册账号的域名列表 |
| SSE | Server-Sent Events，服务器推送事件，用于流式响应 |
| Rate Limit | 速率限制，控制单位时间内的请求数量 |
| Fastify | 高性能 Node.js Web 框架 |
| 风控 | 风险控制，平台防止滥用的检测机制 |

### 9.3 风险提示

| 风险类型 | 描述 | 缓解措施 |
|---------|------|----------|
| 合规风险 | Codex 服务条款可能禁止此类使用 | 仅供个人学习研究，遵守当地法律法规 |
| 账号封禁 | 批量注册可能触发平台风控 | 控制注册频率、使用代理、多域名分散 |
| 服务稳定性 | 官方接口变更可能导致服务中断 | 监控告警、快速切换备用方案 |
| 成本控制 | 域名、接码、服务器均有成本 | ROI 计算、设置预算上限 |
| 安全风险 | 账号凭证泄露可能导致损失 | 加密存储、访问控制、定期审计 |
