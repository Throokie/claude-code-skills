## 2. 产品概述

### 2.1 产品背景
Codex 是 OpenAI 提供的代码生成模型服务，官方 API 存在以下痛点：
- 访问限制：部分地区或场景访问受限
- 成本问题：官方定价对高频使用者成本较高
- 账号管理：单个账号有配额限制，需要多账号轮询

本项目旨在通过自动化账号创建 + 反代技术，构建一个可用的免费 API 池服务。

### 2.2 产品目标
- **P0**: 实现自动创建 Codex 账号（用户提供域名池 + 接码服务）
- **P0**: 集成开源反代项目，将 Codex 接口转换为标准 OpenAI API 格式
- **P0**: 实现多账号轮询调度，提高可用性和配额利用率
- **P1**: 实现账号健康监控与自动替换机制
- **P1**: 提供管理后台，可视化监控 API 池状态
- **P2**: 实现用户配额管理和访问控制

### 2.3 目标用户
| 用户类型 | 特征 | 主要需求 |
|---------|------|---------|
| 开发者 | 需要代码补全/生成功能 | 稳定、低延迟的 API 服务 |
| AI 应用构建者 | 集成 Codex 到自有产品 | 标准 API 格式、高可用性 |
| 研究团队 | 大量使用代码生成能力 | 成本可控、配额充足 |

### 3.2 详细功能描述

#### F-001: 账号自动注册
**用户故事**: 作为管理员，我想要系统自动创建 Codex 账号，以便快速扩充 API 池容量

**验收标准**:
- [ ] 支持配置域名池（用户输入的域名列表）
- [ ] 支持配置接码平台 API（如 5sim、smshub 等）
- [ ] 自动化注册流程：生成邮箱 → 接码获取手机号 → 注册账号 → 保存凭证
- [ ] 注册成功率监控，失败自动重试
- [ ] 支持注册速率限制，防止被封

**输入/输出**:
- 输入: 域名池配置、接码平台配置、注册数量
- 输出: 注册成功的账号列表（保存到数据库）

**边界情况**:
- 接码平台无可用号码: 等待并重试，或切换到备用接码平台
- 注册触发风控: 增加延时、更换 IP 代理、降低频率
- 域名被拉黑: 标记域名不可用，从池中移除

#### F-002: 账号状态管理
**用户故事**: 作为管理员，我想要管理 API 池中的账号，以便维护服务可用性

**验收标准**:
- [ ] CRUD 操作：添加、删除、编辑、查询账号
- [ ] 状态追踪：正常、限速、封禁、失效等状态
- [ ] 配额追踪：记录每个账号的剩余额度
- [ ] 批量导入/导出账号

**账号状态定义**:
| 状态 | 说明 | 自动切换 |
|------|------|----------|
| active | 正常可用 | - |
| rate_limited | 触发速率限制 | 冷却 1 小时后重试 |
| suspended | 被平台封禁 | 自动标记，需人工确认 |
| expired | 额度耗尽 | 自动下线 |
| error | 其他错误 | 重试 3 次后标记 |

#### F-004: 多账号负载均衡
**用户故事**: 作为 API 使用者，我想要高可用的服务，以便在部分账号失效时仍能正常使用

**验收标准**:
- [ ] 支持多种负载均衡策略：轮询、随机、加权、最少连接
- [ ] 健康检查：定期检测账号可用性
- [ ] 故障转移：账号失效时自动切换到可用账号
- [ ] 会话保持：同一用户请求优先路由到同一账号（可选）

**路由策略配置**:
```yaml
load_balance:
  strategy: round_robin  # round_robin | random | weighted | least_conn
  health_check:
    enabled: true
    interval: 60s
    timeout: 10s
  retry:
    max_attempts: 3
    backoff: exponential
```

#### F-006: 健康监控告警
**用户故事**: 作为运维人员，我想要及时发现服务异常，以便快速响应修复

**验收标准**:
- [ ] 账号健康度仪表盘（可用率、响应时间、错误率）
- [ ] 告警机制：账号大面积失效、请求成功率下降、响应时间异常
- [ ] 告警渠道：Webhook、Telegram、Email
- [ ] 自动恢复：部分场景自动尝试恢复（如 rate limit 冷却）

**监控指标**:
- 账号可用数量 / 总数量
- 请求成功率 / 失败率
- 平均响应时间 / P95 / P99
- 每个账号的请求量分布

## 4. 非功能需求

### 4.1 性能要求
- API 响应时间：P95 < 5s（不含 Codex 生成时间）
- 网关吞吐量：支持 100+ QPS
- 账号健康检查周期：60s
- 支持并发请求：支持至少 50 个并发连接

### 4.2 安全要求
- [ ] API Key 安全存储（哈希加密）
- [ ] 请求内容日志脱敏（不记录 prompt 内容）
- [ ] 防止账号凭证泄露（数据库加密存储）
- [ ] 访问日志审计
- [ ] 速率限制防止滥用

### 4.3 兼容性要求
- API 格式兼容 OpenAI API v1
- 支持标准 HTTP/HTTPS 代理配置
- 支持 SSE 流式响应

### 4.4 可靠性要求
- 服务可用性目标：99%（单点故障不影响整体）
- 账号失效自动剔除，无需人工干预
- 支持优雅重启，不中断正在处理的请求

interface Account {
